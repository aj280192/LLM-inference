// CHAPTER 02 — WHY YOUR RESPONSE IS SLOW
THE KV CACHE AND THE BANDWIDTH WALL
Inside the GPU while your API call waits for tokens
From Ch.01: Every output token triggers a full forward pass — reading 140 GB of model weights. But that's not the only cost. There's a second growing cost that makes every pass heavier than the last.
WHY THE MODEL SAVES ANYTHING AT ALL
You asked: "What is the Eiffel Tower and when was it built?"
The model reads your entire question first, then generates each answer word one at a time. For each new word, it must "look back" at your question and every word it has already written — to decide what comes next.

Without saving anything, that means recomputing from scratch each token. So the model stores a compressed record for every token it has seen: K (what that token is about) and V (what it carries) — one K+V pair per transformer layer. This is the KV cache. It avoids recomputation — but fills GPU memory, growing with every token.

// CHAPTER 04 — THE ENGINE FIX
vLLM: PAGEDATTENTION + CONTINUOUS BATCHING
The OS taught us to solve this 50 years ago — vLLM applies it to GPU memory
From Ch.03: The naïve approach wastes ~70% of VRAM with contiguous pre-allocation and serves requests serially. vLLM solves both with two ideas borrowed from operating systems.
PAGEDATTENTION — "OS VIRTUAL MEMORY FOR KV CACHE"
Your OS doesn't give a process one giant contiguous RAM block. It maps virtual addresses to scattered physical pages — and only allocates pages that are actually used. vLLM does the exact same thing to GPU KV cache memory.
BEFORE — contiguous slab per request, fragmented tails
Each row = one request's pre-allocated slab. Dashed = wasted.
AFTER — pages scattered freely across VRAM, allocated on demand
■ A ■ B ■ C ■ D □ free
RESULT
KV waste: 70% → <4%. Pages shared across beam-search branches (copy-on-write). OOM replaced by predictable backpressure. All with zero quality change.
CONTINUOUS BATCHING — "ITERATION-LEVEL SCHEDULING"
Old static batching: batch waits for the slowest request to finish — then accepts new work. Continuous batching: at every single token step, the scheduler can evict finished sequences and inject waiting requests. Zero idle cycles.

// CHAPTER 05 — ATTACKING THE ROOT CAUSE
CUTTING BYTES-PER-TOKEN: QUANT + SPEC DECODE
vLLM fixed memory management. These two techniques attack the bandwidth wall itself.
From Ch.02: The root cause is bytes-read-per-token from HBM. vLLM didn't reduce that — it just stopped wasting VRAM. Now we target the bytes themselves.
QUANTIZATION — SHRINK THE BYTES WE READ
Every token generation reads all model weights from HBM. Fewer bits per weight = fewer bytes on the memory bus per token. FP8 cuts the read in half vs FP16. FP4 cuts it in four. H100 and Blackwell have dedicated silicon for FP8/FP4 — zero emulation overhead.

SPECULATIVE DECODING — BATCH THE FORWARD PASSES
Root cause (again): N tokens = N sequential large-model memory reads. What if we could verify N tokens in one pass? Use a cheap 1B model to guess N tokens. Let the 70B model verify all N in parallel — like prefill.

With spec decode: 5 tokens ≈ 1 draft pass + 1 verify pass = 2–3× lower latency. Same output — the 70B model is the final authority on what gets emitted.
CHAPTER TAKEAWAY — BOTH ATTACK BYTES-PER-TOKEN
Quantisation reads fewer bytes per weight. Spec decoding emits more tokens per large-model read. Together on an H100 with FP8 + speculative decoding: ~4–5× throughput gain over FP16 naive.

Scaling from one pod to a fleet introduces a completely different class of problems
